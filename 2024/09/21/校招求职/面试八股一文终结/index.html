    计算机网络

    TCP/IP

    • 层次模型
      • 应用层：规定了具体应用的处理协议。如 HTTP，FTP，DNS
      • 传输层：为应用层提供连接支持，如 TCP，UDP
      • 网络层：实现点对点的通信，为数据规划合理路径。IP
      • 链路层：用来处理连接网络的硬件部分。主要是网卡驱动完成
    • 协议簇

    • TCP UDP 对比
      • 有无连接：TCP 先通过三次握手，建立连接，传输过程中进行流量控制和拥塞控制，保证数据的顺序性和及时性，传输结束之后四次挥手结束连接。而 UDP 是一种无连接协议，直接发送报文，不进行三次握手四次挥手，UDP 速度快，但是不提供数据的可靠性和完整性保证，数据报文可能会丢失或乱序。
      • 可靠性：TCP 在传输过程中会对应用层报文进行分段，重传，确认，可以保证数据的完整性和可靠性(校验机制）。
      • 适用场景：TCP 由于提供可靠传输，适用于大多数场景，文件传输，网页浏览，电子邮件。UDP 适用于速度要求高，但是可靠性要求不那么高的情景：游戏，音视频，实时通信
      • 头部结构：TCP 负责的头部，UDP 只有 8 字节头部

    HTTP

    • 状态码
      • 1xx 类状态码属于提示信息，是协议处理中的一种中间状态，实际用到的比较少。
      • 2xx 类状态码表示服务器成功处理了客户端的请求，也是我们最愿意看到的状态。
      • 3xx 类状态码表示所请求资源发生变动，需要用新的 URL 进行请求，也就是重定向
      • 4xx 类状态码表示客户端发送的报文有误，服务器无法处理，也就是错误码的含义。
      • 5xx 类状态码表示客户端请求报文正确，但是服务器处理时内部发生了错误，属于服务器端的错误码。
    • GET 和 POST
      • GET 的语义是请求获取指定的资源。GET 方法是安全、幂等、可被缓存的。
      • POST 的语义是根据请求负荷（报文主体）对指定的资源做出处理，具体的处理方式视资源类型而不同。POST 不安全，不幂等，（大部分实现）不可缓存。
    • 从URL 到显示页面的过程
      • DNS 将URL转IP
      • 有了服务端IP,发送HTTP连接请求，TCP 开始建立连接
      • 服务器将 html 传回主机，主机交给浏览器进行解析渲染

    TCP

    有连接，可靠，基于字节流

    • 三次握手
      • 过程
        • 发送端首先发送一个带 SYN 标志的数据包给对方。
        • 接收端收到后，回传一个带有 SYN/ACK 标志的数据包以示传达确认信息。
        • 最后，发送端再回传一个带 ACK 标志的数据包，代表“握手”结束
      • 为什么要三次握手，两次不行？
        • 主要是为了在防止在网络拥塞情况下，服务端资源浪费。
        • 具体来说，假如是两次握手，意味着服务端分配资源建立连接的时机应该是收到客户端的第一次握手立即建立。然而假如网络拥塞导致客户端发送了两个连接请求，就意味着服务端要建立两个连接。
        • 采用三次握手就可以在客户端第二次确认连接的时候，服务端才分配资源建立连接。
    • 四次挥手
      • 过程
        • 客户端发送 FIN
        • 服务端收到后进入 close-wait 状态，回送客户端 ACK,此时连接进入半关闭，服务端 将未尽事宜处理完，发送给客户端一个 FIN, 自己进入 last-ack 状态。
        • 客户端收到最后的 FIN,回复 ACK ,并设置 2MSL 的等待时间。之后关闭连接
        • 服务端收到客户端的 ACK 关闭连接
      • 为什么客户端要 2MSL
        • 防止客户端发出去的 ACK 没收到，服务端又发起关闭连接请求。
    • 可靠传输
      • 通过超时重传机制实现，超时标准为加权平均往返时间 RTTs，RTTs 的更新机制是新老往返时间按比例加权
    • 滑动窗口
      • 发送方，接收方都维护一个窗口，当窗口左边确认处理完之后，就可以向右移动到第一个没有确认处理的位置。
    • 流量控制
      • 接收方用于控制发送方的发送速率，通过确认报文中的窗口字段完成，为 0 表示不让发送
    • 拥塞控制

      • 流量控制是为了让接收方能来得及接收，而拥塞控制是为了降低整个网络的拥塞程度。
      • 主要通过 慢开始、拥塞避免、快重传、快恢复 机制实现
      • 要明白上面四个机制，首先明白 控制窗口 cwnd 和门限 ssthresh 的概念，前者是动态变化，后者是阶段性设置
      • 慢开始：从 1 开始， 指数增长
      • 拥塞避免：达到门限后线性增长
      • 快重传和快恢复：遇到三次连续 ack 时执行，门限和控制窗口同时减半然后线性增长。此时认为并不是网络拥塞，而是个别丢包。
      • 超时被认为是网络拥塞，采用慢开始策略。

    • 粘包

      • 由于TCP 是面向字节流的，会对上层数据进行拆分。自然的就会发生将上层不同消息的数据放到一个TCP报文的情况，这就叫刚发生了粘包
      • 解决方法：靠应用层规定好消息的边界。具体的可以规定数据结构，可以用字段专门制定数据大小。

    IP

    • IP 分类地址
      • A 类、B 类、C 类、D 类、E 类。黄色为类号
      • ABC类中最大主机数为 2^n - 2, n 为主机号位数。减去的两个为主机号全零用于指代网络号，全一用于广播。
      • 广播：分本网络内广播(路由器不转发，本网络内所有主机由于通过以太网直接连接，一定会接受到广播报文)，跨网络广播(一般情况下，出于安全考虑，路由器不转发这类报文。)
      • 组播：将包发送给特定组内(通过组播号区分)的所有主机。组播 = 多播；
      • 单播，广播，组播 并列概念
    • IP 分类地址的缺点
      • 类下面没有层次，如果一个公司用了 B 类地址，无法根据部门关系分配 IP
      • 网络大小是给死的，不灵活：C 类 254 个 IP, B 类 6万多个，大部分公司被迫申请 B 却用不完，造成浪费。
    • 无分类地址 CIDR
      • 没有ABC类的概念，采用前缀长度标记网络号的位数。
    • IP 协议相关技术
      • DNS 协议解析过程

      • ARP
        • ip 地址指示了目的地是哪一个，但是每一步应该给局域网中哪个节点就需要 mac 地址，如果当前的节点不知道目的 ip 对应的 mac 地址，就需要用 arp 协议解析。
      • ICMP 上层协议。包括差错报告和询问。
        • ping 实质是发送一个 icmp 询问报文，对方回答之后，计算往返时间。
        • Traceroute 通过发送一系列具有不同 ttl 值的 UDP 报文，使得沿途每一个路由节点回送超时的抄错报告报文，从而获得每个路由节点。

    • 虚拟专用网 VPN

      • 通过互联网搭建虚拟的内部局域网。发送端通过vpn软件在加密报文，然后通过公网ip上隧道进行传输。接收端接收后解密，然后更改目的 ip。

    • 网络地址转换 NAT 技术

      • 内部局域网之间用私有地址，访问外部才转换为公网地址。
